결제는 "성공 아니면 실패"처럼 보이지만, 실제로는 그 사이에 수많은 중간 상태가 있습니다. 승인 요청은 보냈는데 응답을 못 받은 경우, 승인은 됐는데 취소가 실패한 경우 — 이런 경계를 어떻게 다루느냐가 안정성을 가릅니다.

## 상태를 먼저 정의한다

코드를 짜기 전에 결제가 가질 수 있는 상태를 명시적으로 적었습니다.

```ts
type PaymentStatus =
  | "REQUESTED" // 승인 요청 보냄
  | "APPROVED" // 승인 완료
  | "CANCEL_REQUESTED" // 취소 요청 보냄
  | "CANCELED" // 취소 완료
  | "FAILED"; // 실패
```

## 멱등성이 핵심

같은 요청이 두 번 들어와도 결과가 한 번만 반영되도록, 거래마다 고유 키를 두고 중복 승인을 막았습니다. 네트워크가 불안정한 결제 환경에서 멱등성은 선택이 아니라 필수입니다.

## 정리

결제 연동의 어려움은 "정상 흐름"이 아니라 "어긋난 흐름"에 있습니다. 상태를 먼저 그리고, 각 전이를 멱등하게 만들면 운영 환경에서도 흔들리지 않습니다.
